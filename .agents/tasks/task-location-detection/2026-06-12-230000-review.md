# 3-tier location detection via Places365, CLIP, and GeoEstimation

A new `LocationAnalyzer` adds scene classification (Places365 ResNet-50), landmark detection (CLIP cosine similarity against a 50-entry database), and coarse geographic region estimation (CLIP similarity + Nominatim reverse geocoding) to the analysis pipeline. The test page gains a dropdown filter that queries the existing `/api/photos?categories=` endpoint. The approach is sound for a first pass — CLIP cosine similarity is a reasonable zero-shot signal — but several issues threaten reliability on real uploads, especially around model downloads at runtime, unbounded network calls, and threshold calibration.

**Watch for:** CLIP cosine similarities are not probabilities and the 0.25 threshold will almost certainly fire false positives on arbitrary photos (confirmed). Nominatim reverse geocoding is called synchronously per region match with no timeout or rate-limiting, risking request hangs on upload (confirmed). Places365 weights are downloaded over plain HTTP from an external server the first time a user uploads a photo, blocking the request for hundreds of MB with no timeout (confirmed).

## High-level view

The three detection tiers share a single CLIP model but differ in calibration risk. Places365 uses proper softmax probabilities and a conservative threshold; CLIP landmark and geo tiers use raw cosine similarity scores as if they were calibrated confidence, which will produce false matches on most photos since CLIP similarity between an image and *any* text prompt tends to hover around 0.20–0.30.

Model loading happens lazily on the first request if warmup fails (or was never called), and the Places365 download uses `urllib.request.urlretrieve` over plain HTTP with no timeout, no progress feedback, and no integrity check. A first-upload experience on a fresh deployment will block for minutes while downloading ~100 MB of weights.

The geo-estimation tier calls Nominatim for every region above threshold — potentially all 11 regions — on every photo upload, with no timeout or caching. Nominatim's usage policy limits requests to 1/second, so bursts will get the app's user-agent banned.

The test page filter integrates cleanly with the existing `/api/photos?categories=` and `/api/tags` APIs. The only behavioral concern is the "no photos" empty state message changing to reference filtering even on first load when no filter is active.

1. **CLIP threshold miscalibration** — Cosine similarity 0.25 is not a meaningful confidence boundary for CLIP ViT-B-32; most image-text pairs score 0.20–0.35. Either raise the threshold to ~0.35+ or apply softmax over the candidate set to get a proper distribution. (confirmed)

3. **Blocking model download on first request** — `urllib.request.urlretrieve` over HTTP with no timeout; if the server is slow the upload request hangs indefinitely. Add a timeout, validate the file hash after download, and use HTTPS where available. (confirmed)
4. **Confidence values semantically misleading** — `TagResult.confidence` from the landmark/geo tiers stores cosine similarity (0–1 range but not probability); downstream consumers and the UI treat it as a probability. Normalize through softmax or clearly document the discrepancy. (confirmed)
5. **`weights_only=False` in torch.load** — Enables arbitrary code execution from the checkpoint file. Since the file is downloaded from an HTTP URL without integrity verification, this is a supply-chain risk. Pin a SHA-256 hash and verify before loading. (confirmed)
6. **Empty-state message regression** — Gallery shows "No photos match the current filter" even when no filter is active and the database is genuinely empty. The original empty-state message was replaced unconditionally. (confirmed)

## CLIP similarity is not confidence

The landmark detector computes `image_features @ landmark_text_features.T` and treats raw cosine similarity scores as confidence values stored in `TagResult.confidence`. For CLIP ViT-B-32 (laion2b), cosine similarity between an image and an arbitrary text prompt typically falls in the 0.20–0.35 range. The threshold of 0.25 means most photos of any outdoor scene will match multiple landmarks. A photo of a generic bridge will likely match "Golden Gate Bridge", "Tower Bridge", and "Charles Bridge" simultaneously.

The same issue applies to `_estimate_geo` with a threshold of 0.20 — nearly every image will match several of the 11 regions, triggering multiple reverse-geocoding calls.

The fix is to apply softmax over the candidate set (as Places365 correctly does) and use the resulting distribution to pick top-k, or to raise the raw-similarity threshold substantially (empirically ~0.32–0.35 for meaningful matches with this model).

## Nominatim rate-limiting and timeout

`_reverse_geocode` creates a new `Nominatim` geolocator instance on every call, issues a blocking `geolocator.reverse()` wrapped in `asyncio.to_thread`, and has no timeout parameter. Nominatim's terms of use require at most 1 request/second from a single user-agent. When `_estimate_geo` fires for multiple regions (likely given the low threshold), the app sends a burst of geocode requests in rapid succession. This will:

1. Block the upload response for seconds per call (Nominatim can be slow)
2. Get the `"photo-tagger"` user-agent rate-limited or banned

Since the 11 region centroids never change, these results should be cached at startup or pre-computed as static data. At minimum, pass `timeout=5` to the Nominatim constructor.

## Model download at request time

`_load_places365` downloads ~100 MB from `http://places2.csail.mit.edu/` using `urllib.request.urlretrieve` with no timeout. If warmup fails silently (the pipeline catches the exception and logs a warning), the first user upload triggers this download synchronously. The HTTP URL also means the download is unauthenticated and unencrypted — a MitM could serve a malicious checkpoint that `torch.load(..., weights_only=False)` would execute.

Recommendations: download models in a build step or Docker layer, verify SHA-256 after download, switch to `weights_only=True` or `torch.load` with `torch.serialization.add_safe_globals()`.

## Overlap with existing SceneAnalyzer

Both `SceneAnalyzer` and `LocationAnalyzer` produce `WHERE` tags with partially overlapping vocabularies (e.g., "beach"). The pipeline's `_deduplicate` keeps the higher-confidence version per `(category, value)` pair. This means Places365's scene predictions are effectively shadowed whenever the rule-based SceneAnalyzer matches the same label with higher confidence — a user won't see the ML-based "beach" at 0.4 if the rule-based system already produced "beach" at 0.75. Whether this shadowing is intentional or an oversight depends on intended tag semantics; if Places365 is meant to provide richer scene context, its labels should be prefixed or use a distinct sub-category to avoid silent deduplication.

## Empty-state message change

Line 569 in test.html unconditionally shows "No photos match the current filter. Try a different tag or clear the filter." This message now appears even when `currentFilter` is null and the database is empty. The original message ("No photos uploaded yet. Upload one above to get started!") should be preserved for the no-filter case:

```javascript
if (!data.photos || data.photos.length === 0) {
    const msg = currentFilter
        ? 'No photos match the current filter. Try a different tag or clear the filter.'
        : 'No photos uploaded yet. Upload one above to get started!';
    galleryContainer.innerHTML = `<div class="gallery-empty">${msg}</div>`;
    return;
}
```
