# ML-powered photo tagger with upload, analysis, and multi-category filtering

A full-stack photo tagging application: FastAPI backend with async SQLAlchemy persistence, a three-stage ML pipeline (DeepFace for faces, YOLOv8n for objects, rule-based scene inference), and a Next.js + TypeScript frontend with a responsive grid, multi-category filter panel, and bounding-box overlay in the detail view. The end-to-end flow—upload → analyze → store → display → filter—works as a single synchronous request, meaning large images block the response until all three ML models complete.

Watch for:
- Upload endpoint runs full ML analysis synchronously, blocking the HTTP response for potentially many seconds per photo (confirmed).
- Filepath stored in `PhotoResponse` is leaked to the frontend, exposing the server's internal directory structure (confirmed).
- Bounding box coordinate systems differ between backend (pixel values) and frontend (percentage-based rendering), so overlay boxes will be mispositioned (confirmed).
- `sort_by` field from the query string is passed directly to `getattr(Photo, sort_by)`, allowing attribute probing on the model (confirmed).
- SceneAnalyzer creates a second YOLO model instance per request instead of reusing the one from ObjectAnalyzer (confirmed).

## High-level view

The backend follows a clean layered architecture—routes, service layer, analyzers—with proper async session management and Pydantic validation at the API boundary. The ML pipeline is composable: each analyzer implements a `BaseAnalyzer` ABC and the pipeline deduplicates by (category, value). The weakness is that analysis runs inline with the upload request, so a single large photo with all three analyzers can easily take 5–15 seconds before the client receives a response.

The frontend uses React Query for server state and keeps UI logic in well-separated components. Filter state is lifted to the page and serialized to JSON for the backend. The primary data contract mismatch is around bounding boxes: the backend emits pixel coordinates while `PhotoDetail.tsx` renders them as percentage offsets, guaranteeing incorrect overlay placement.

Security exposure is minimal for a local-only prototype, but the API leaks the server's absolute filesystem path in every photo response and accepts arbitrary `sort_by` values that resolve to any attribute on the `Photo` model—including `filepath`. CORS is locked to `localhost:3000`, which is appropriate for development but would need tightening for any deployment.

Test coverage is solid for the happy path through the API (upload, list, filter, delete) and model layer, but all API-level tests run against a mock pipeline, so there are no tests exercising real analyzer output parsing or error recovery.

1. **Synchronous analysis on upload** — Move ML analysis to a background task (or return immediately with `analyzed=false` and poll/websocket for completion) to avoid multi-second blocking responses.
2. **Filepath exposure in API response** — Remove `filepath` from `PhotoResponse` or replace it with a relative URL. The frontend already constructs image URLs via `getImageUrl(filename)` and doesn't use `filepath`.

6. **No file-content validation** — Extension check occurs on the filename, but content bytes are never validated as a real image before being written to disk. A file named `evil.jpg` with arbitrary content passes.
7. **ObjectAnalyzer emits duplicate tags** — The deduplication in the pipeline keeps one per (category, value), but ObjectAnalyzer itself appends a tag per bounding box before deduplication, generating arbitrarily many intermediate `TagResult` objects for repeated object classes.
8. **Frontend sort values don't map to backend columns** — `SortBy.DATE` sends `"date"` but the backend column is `upload_date`; `SortBy.NUM_PEOPLE` and `SortBy.CONFIDENCE` have no backend column equivalent and silently fall back to `upload_date`.

## Synchronous upload-then-analyze pipeline

The upload endpoint in `photos.py` calls `photo_service.save_upload` then immediately calls `photo_service.analyze_photo`, which runs all three ML analyzers sequentially before returning the HTTP response. For a moderately sized image, DeepFace face analysis alone can take 2–4 seconds, and YOLO inference adds another 1–2 seconds—executed twice because `SceneAnalyzer` loads its own YOLO model independently of `ObjectAnalyzer`.

This means the frontend's `UploadModal` shows "Analyzing..." for the entire duration and, critically, the `uploadPhoto` fetch call will time out or hang for multi-second stretches. For a production system this should be a background task with status polling; even for a prototype it limits batch upload usability since `processFile` in `UploadModal.tsx` fires uploads sequentially with only a 100ms stagger.

## Filesystem path leakage

`PhotoResponse` includes `filepath: str` which contains the full server-side path (e.g. `./uploads/abc123def.jpg` or the absolute resolved path). This is transmitted to every frontend consumer. The frontend never uses this field—it reconstructs the image URL via `getImageUrl(photo.filename)`. The field should be removed from the response schema or replaced with a relative URL path.

## Bounding box coordinate system mismatch

In `object_analyzer.py` and `face_analyzer.py`, bounding boxes are serialized as pixel coordinates:

```python
bbox = json.dumps({
    "x": int(xyxy[0]),  # pixels
    "y": int(xyxy[1]),  # pixels
    "w": int(xyxy[2] - xyxy[0]),
    "h": int(xyxy[3] - xyxy[1]),
})
```

But in `PhotoDetail.tsx`, the overlay renders them as percentages:

```tsx
style={{
  left: `${bbox.x * 100}%`,
  top: `${bbox.y * 100}%`,
  width: `${bbox.width * 100}%`,
  height: `${bbox.height * 100}%`,
}}
```

A bounding box at pixel (200, 150) with the frontend formula becomes `left: 20000%`. Additionally, the frontend `BoundingBox` interface uses `width`/`height` but the backend emits `w`/`h`, so even the property names don't match—`bbox.width` will be `undefined`.

## Unsanitized sort column resolution

In `photo_service.get_photos`:

```python
sort_column = getattr(Photo, sort_by, Photo.upload_date)
```

Any attribute name on the `Photo` ORM class—including `filepath`, `tags`, relationship descriptors—can be reached by passing it as `sort_by`. While this won't leak data directly (it's used in an ORDER BY clause), it can trigger SQLAlchemy errors or expose model internals through error messages. An allowlist is the standard fix:

```python
ALLOWED_SORT_FIELDS = {"upload_date", "file_size", "original_filename"}
sort_column = getattr(Photo, sort_by) if sort_by in ALLOWED_SORT_FIELDS else Photo.upload_date
```

## Frontend-backend sort field mismatch

The frontend `SortBy` enum sends these values: `"date"`, `"num_people"`, `"location"`, `"confidence"`. The backend resolves `sort_by` through `getattr(Photo, sort_by, Photo.upload_date)`:

- `"date"` → no such attribute → silent fallback to `upload_date`
- `"num_people"` → no such attribute → silent fallback
- `"location"` → no such attribute → silent fallback
- `"confidence"` → no such attribute → silent fallback

Every sort option except the default is non-functional. The frontend gives the user four choices; only "Date" (accidentally) works.

## Duplicate YOLO instantiation in SceneAnalyzer

`ObjectAnalyzer` caches its YOLO model in `self._model` and reuses it across requests. `SceneAnalyzer`, however, does `YOLO("yolov8n.pt")` inside its `analyze` method on every call:

```python
async def analyze(self, image_path: str) -> list[TagResult]:
    ...
    model = YOLO("yolov8n.pt")
    results = model(image_path, verbose=False)
```

This means every photo upload loads the YOLO model twice and runs inference twice on the same image. The scene analyzer should either accept object detection results from the pipeline (since it only needs the set of detected class names) or at minimum cache the model instance.

## File content not validated

The upload endpoint checks the file extension but never verifies the content is actually a valid image before writing it to disk. `Pillow` is used afterward for dimension extraction, but its failure is silently caught:

```python
try:
    with Image.open(filepath) as img:
        width, height = img.size
except Exception:
    pass
```

An attacker (or even a user accidentally renaming a file) could upload arbitrary content as `.jpg`. The file gets written, a database record is created, and ML analyzers will fail silently. Adding `Image.verify()` before persisting would catch malformed files.

## Test coverage gaps

The test suite mocks the entire analysis pipeline, so no test ever verifies that `FaceAnalyzer`, `ObjectAnalyzer`, or `SceneAnalyzer` parse model output correctly, handle edge cases (no faces, YOLO returning zero boxes), or produce valid `TagResult` objects. The integration tests use the same mock pipeline as the unit tests.

There are no frontend tests. The TypeScript compilation during `pnpm build` serves as a basic type-safety check, but there's no testing of filter logic, upload state machine transitions, or the bounding-box rendering math.
